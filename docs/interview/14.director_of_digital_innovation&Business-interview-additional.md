### Q1 — System Design (Live Whiteboard)

**What to expect:** A panel or the Director asks you to design a system on the spot.
Common prompts: KHQR payment API, a notification system, an idempotency layer.

**Always follow this five-step structure:**

**1 — Clarify constraints (spend 2 minutes here)**

> "Before I design anything, I want to make sure I understand the problem correctly.
> What's the expected transaction volume at peak — requests per second? What's the SLA on response time?
> Do we need KHR and USD support? Is this internal only, or will third-party merchants consume it?
> What's the failure tolerance — can we be eventually consistent, or does every write need to be synchronous?"

Never draw a single box until you've asked these questions. This alone signals senior-level thinking.

**2 — Define the data model**

> "The core entity is a `Payment` record: `PaymentID` (UUID), `SenderAccountID`, `ReceiverAccountID`,
> `Amount`, `Currency`, `Status` (PENDING → PROCESSING → COMPLETED / FAILED),
> `IdempotencyKey`, `CreatedAt`, `UpdatedAt`.
> The `IdempotencyKey` is the most important field — it's what prevents duplicate charges."

**3 — Design the API contract**

```
POST /v1/payments
Headers:
  Idempotency-Key: <client-generated UUID>

Body:
  { "sender_account": "...", "receiver_account": "...", "amount": 100.00, "currency": "USD" }

Responses:
  201 Created  → { "payment_id": "...", "status": "PENDING" }
  409 Conflict → { "error": "duplicate_request", "payment_id": "..." }   ← same idempotent response
  422          → { "error": "insufficient_funds", "message": "..." }
```

**4 — Walk through happy path and failure cases**

| Path | Behaviour |
|------|-----------|
| Happy path | Validate → debit sender → credit receiver → notify → return `COMPLETED` |
| Flexcube timeout | Return `PENDING`, let client poll `GET /v1/payments/{id}` |
| Flexcube error | Return `FAILED` with a specific error code — never `"Something went wrong"` |
| Duplicate request | Return cached response from Redis — Flexcube never sees the retry |

**5 — Address non-functional requirements**

| Concern | Solution |
|---------|----------|
| Idempotency | Store `IdempotencyKey` in Redis with 24-hour TTL |
| Consistency | Saga pattern or two-phase commit for distributed transactions |
| Performance | Async processing via Kafka for non-critical steps (notifications, ledger updates) |
| Observability | Correlation ID propagated through every service — every hop traceable |

> **The interviewer wants to see how you think, not just the final answer. Think out loud at every step.**

### Q2 — How do you design for 10x your current traffic?

> "I design systems to scale horizontally from day one, so that scaling becomes an operational decision — not a re-architecture crisis.
>
> When facing a 10x spike — payday, a promotional campaign, or a regulatory deadline — my approach is always the same: **identify the bottleneck first**. You cannot optimize what you haven't measured.
>
> At the API layer, stateless services scale horizontally behind a load balancer. Auto-scaling handles this automatically.
>
> At the database layer, read replicas absorb read-heavy traffic. For balance queries, I serve from Redis with a 30–60 second TTL — the vast majority of customers don't need real-time balance for a page load, and the database doesn't need to answer 50,000 identical requests.
>
> The most important principle for banking at peak load: **decouple the customer-facing response from the backend processing**. When Flexcube can't absorb 10x calls, I put a queue in front of it. The customer receives an immediate `PENDING` response. Flexcube processes at its own rate. The result comes back via push notification. The app feels fast even when the backend is under pressure."

| Layer | Scaling Strategy |
|-------|-----------------|
| API servers | Horizontal scaling, auto-scaling groups |
| Database reads | Read replicas + Redis cache (30–60s TTL for balances) |
| Database writes | Connection pooling, write-ahead log, batch inserts |
| Downstream (Flexcube) | Async queue (Kafka), circuit breakers, bulkheads |
| Static assets | CDN |

### Q3 — How do you lead and mentor junior developers?

> "My goal as a senior engineer is to make myself less necessary over time — not more indispensable.
>
> **Code reviews are my most powerful teaching tool.** I don't just approve or reject — I explain the reasoning. 'This endpoint doesn't handle a Flexcube timeout. Here's how I'd add a circuit breaker and what happens to the user if we don't.' One comment like that teaches more than an hour-long lecture.
>
> **When someone is stuck, I guide rather than solve.** I sit with them and ask: 'What have you tried? What does the error log tell you?' I want them to reach the answer — not to watch me reach it for them.
>
> **I give juniors ownership of real features, not just bug fixes.** I define clear acceptance criteria upfront and review the architecture before a line of code is written. That catches design problems early — not at demo time, not in production."

> Directors hire senior engineers to multiply the team's output, not just their own. Show you understand that.

### Q4 — Tell me about a time you delivered under an extremely tight deadline

> "We had a regulatory requirement — the NBC mandated a new transaction reporting format within six weeks. The spec arrived at week zero.
>
> The first thing I did was a scope triage: what is mandatory for compliance versus what is desirable? The mandatory piece was the reporting API and the automated file submission mechanism. The internal dashboard was not. I deferred it explicitly.
>
> I split the work into two parallel tracks — one engineer owned the data pipeline, I owned the API and submission layer. We ran daily check-ins: 15 minutes, blockers only. If something was going to slip, I wanted to know within 24 hours, not at the end of the sprint.
>
> We delivered the compliant system in five weeks. The dashboard shipped in the following sprint with no pressure and no shortcuts.
>
> The lesson I took from that: **deadlines should compress scope, never correctness on the critical path.** What you're willing to cut is a product decision. How correctly the remaining scope works is not negotiable."

### Q5 — How do you ensure your systems are observable?

> "In banking, 'I can't tell what's happening' is not an acceptable state. If you can't see your system clearly, you cannot fix it under pressure — and in a regulated environment, pressure is the baseline.
>
> I implement the three pillars of observability, and I'm deliberate about each one."

| Pillar | Implementation |
|--------|---------------|
| **Logs** | Structured JSON with correlation IDs — every request traceable end-to-end across services |
| **Metrics** | Business metrics first: transactions/second, payment failure rate, P99 latency — not just CPU and RAM |
| **Traces** | Distributed tracing via OpenTelemetry — pinpoint exactly where a 2-second request spent its time |

> "Beyond the three pillars, I define **SLIs and SLOs** with the business before launch:
> - **SLI** (what we measure): payment API P99 latency
> - **SLO** (the target): P99 < 500ms, 99.9% of the time
> - **Alert**: fires when we're consuming the error budget faster than the agreed burn rate
>
> The metric I care most about is not technical health — it's business health. Failed payment rate, duplicate transaction count, KYC drop-off rate. Those are the numbers a Director actually cares about."

### Q6 — What are your API design principles?

> "I treat APIs as public contracts. Once a consumer depends on your API, a breaking change is not a technical decision — it's a business and trust decision. I design with that weight in mind from day one."

| Principle | How I Apply It |
|-----------|---------------|
| **Versioning** | `/v1/payments` — never mutate the contract, add `/v2` when behaviour changes |
| **Idempotency** | Every write endpoint accepts an `Idempotency-Key` header. Non-negotiable for payments. |
| **Meaningful errors** | `{ "error": "insufficient_funds", "message": "Account balance below transfer amount" }` — never `500 Internal Error` |
| **Pagination** | Cursor-based for large result sets. Offset pagination breaks under concurrent writes. |
| **Backward compatibility** | Add fields freely. Never remove or rename. Old clients must continue to work. |
| **Rate limiting** | Per-user, per-endpoint. Return `429` with a `Retry-After` header — never a silent failure. |

> "One thing I enforce strictly: **error codes are a product decision, not a backend detail.** The error code returned when a payment fails determines whether the customer retries, calls support, or closes their account. I work with product to define the full error taxonomy before the first endpoint is built."

### Q7 — Walk me through your CI/CD pipeline

> "My philosophy: every stage in the pipeline must either catch a real bug or it has no right to exist. Slow pipelines get bypassed. Bypassed pipelines cause incidents."

```
Developer pushes code
        ↓
Unit tests                    < 2 minutes — fail fast, fail early
        ↓
Static analysis + SAST        OWASP dependency check, SonarQube
        ↓
Integration tests             Against a real database — never mocks
        ↓
Build Docker image → push to registry
        ↓
Deploy to Staging → smoke tests
        ↓
Migration validation          ← Run against a production-size data copy
        ↓
Manual approval gate          For production deployments
        ↓
Blue-green deploy to Production
        ↓
Automated rollback            If error rate spikes within 10 minutes
```

> "The stage most teams skip — and the one that has caused the worst outages I've seen — is **migration validation against production-size data**. A migration that completes in 2 seconds on a 1,000-row staging database can lock a 10-million-row production table for 45 minutes. I learned that lesson early, and I've never skipped that stage since."

### Q8 — How do you manage stakeholder expectations when timelines slip?

> "The worst thing you can do is stay silent and hope it resolves itself. The second worst is giving an optimistic timeline you know you can't defend.
>
> The moment I know a deadline is at risk — not when it's confirmed — I notify the stakeholder. Not 'we might have a problem', but: 'We discovered a dependency on Flexcube's SOAP API that wasn't in the original scope. This adds five days. Here are three options.'
>
> **Option A:** Ship on the original date with the impacted feature descoped.
> **Option B:** Ship the full scope five days late.
> **Option C:** Deliver a manual workaround on time, automate it in the next sprint.
>
> Then I stop talking and let them choose. When they do, I confirm the decision in writing — email or ticket — with explicit scope listed. This protects both sides and removes ambiguity about what 'done' means."

> Directors promote engineers who give them options. Not engineers who drop problems on their desk and leave.

### Q9 — How do you approach a build vs. buy decision?

> "I never evaluate this on license cost alone. The real question is total cost of ownership: integration time, ongoing maintenance, vendor lock-in risk, data model fit, and the opportunity cost of not shipping other features while your team fights with someone else's abstraction.
>
> **Buy when:** the problem is commodity, integration is low-friction, and it's not a core competency. PDF generation, notification delivery, OCR for KYC documents — buy these. Don't build what you don't have to.
>
> **Build when:** the problem is core to your competitive differentiation, the vendor's model doesn't map to your domain, or vendor lock-in creates existential risk. The payment routing logic that defines how KHQR flows through CMB's platform — build that.
>
> **The trap I watch for:** a product that requires so much customisation it becomes a build anyway — but now with the vendor's constraints layered on top. My test: 'Can we use 80% of this out of the box, or are we buying a foundation to build our own product on?' If it's the latter, we build."

### Q10 — How do you handle payment reconciliation?

> "Reconciliation is one of the most underestimated problems in digital banking. It's invisible when it works. When it doesn't, it's a regulatory event.
>
> I implement three layers:
>
> **Real-time:** every transaction that enters my middleware is written to a reconciliation log with a status and timestamp. Any transaction stuck in `PROCESSING` for more than 60 seconds triggers an alert — not a report, an alert.
>
> **End-of-day batch:** we compare our internal ledger against the Flexcube ledger and the Bakong interbank settlement file for KHQR transactions. Every discrepancy is flagged — never silently corrected. Corrections require explicit authorization and are logged with the reason.
>
> **Immutable audit trail:** every reconciliation run is stored permanently. When the NBC asks why a $5,000 discrepancy was adjusted on a Tuesday, I show the full chain of events: who flagged it, who authorized the correction, when it was applied, and the before/after ledger state.
>
> The non-negotiable principle: **you never silently correct a discrepancy in banking.** Every adjustment is explainable, logged, and authorized — or it didn't happen correctly."

### Q11 — How would you design a fraud detection system?

> "Fraud is a layered problem. No single mechanism catches everything, and any single mechanism that tries will either miss fraud or block legitimate customers — both of which are expensive."

**Three layers working together:**

**Layer 1 — Rules engine (real-time, < 50ms)**
Hard deterministic rules. 'Transaction over $10,000 requires secondary verification.' 'Three failed PINs in 60 seconds — lock the card.' These are fast, auditable, and explainable to regulators.

**Layer 2 — ML scoring (near real-time, < 200ms)**
A model trained on historical fraud patterns scores every transaction. A high score triggers step-up authentication (OTP), not an automatic block. You cannot block a legitimate high-value payment based on a probability score alone.

**Layer 3 — Human review (async)**
High-risk accounts and unusual pattern clusters are flagged for the fraud team to review within 24 hours. Not every decision should be automated.

> "The architectural principle I'm firm about: **separate detection from action.** The detection engine outputs a risk score. A separate policy engine decides what to do with it — block, challenge, or allow. This means compliance can update enforcement policy without touching the ML model, and the ML team can retrain without breaking the enforcement rules."

### Q12 — How do you ensure data isolation in a multi-tenant system?

> "In a banking context, multi-tenancy typically means serving multiple corporate clients — agent banking partners, business banking customers — from a shared platform. The isolation model I choose depends on the sensitivity of the data and the regulatory requirement."

| Model | Use Case | Risk |
|-------|----------|------|
| Shared DB, separate schema | Low-risk data, cost-sensitive | A bug can accidentally query wrong schema |
| Separate DB per tenant | Banking-grade isolation | Higher ops overhead — worth it |
| Row-level security (RLS) | Simplest, lowest cost | Every query must enforce `WHERE tenant_id = ?` — no exceptions |

> "For corporate banking clients, I default to **database-per-tenant**. I enforce this at the connection pool level: each tenant's service worker connects to its own database with its own credentials. A bug in Tenant A's query path is physically incapable of reading Tenant B's data. That's the level of isolation banking requires."

### Q13 — How do you prioritize when everything is marked urgent?

> "When everything is urgent, nothing is. My job is to create an actual priority order — not to validate everyone's feeling of urgency.
>
> I use two axes: **Impact vs. Effort**. High impact, low effort ships first. Low impact, high effort goes to the backlog unless there's a specific reason it exists.
>
> In banking, there's a mandatory third axis: **regulatory risk.** A compliance deadline always trumps a feature request, regardless of how loudly stakeholders push.
>
> The most effective tool I've found: **a public priority stack**, visible to all stakeholders. When a new 'urgent' request arrives, I ask one question: 'Which item currently on this list do you want me to push down to make room for this?' That question forces a real prioritization conversation within 60 seconds."

### Q14 — How do you influence decisions when you don't have direct authority?

> "The best technical decisions rarely come from the person with the highest title. Influence without authority is a core skill — especially in large organizations.
>
> **Lead with data, not opinion.** 'I think we should use Kafka' loses to 'I ran a benchmark — RabbitMQ saturates at 5,000 messages/second and we need 20,000 at payday. Here's the test.' Data makes it a different conversation.
>
> **Translate risk into business language.** Engineers see technical risk. Stakeholders see business risk. 'If we skip the caching layer, on the 25th of the month the balance API will receive 50,000 requests in 10 minutes and the database will go down' is a sentence a Director understands and acts on.
>
> **Build support before the room.** If I need organizational momentum, I brief one key stakeholder privately before the formal decision meeting. An idea that enters a room with one visible supporter lands completely differently from one launched cold."

### Q15 — Describe your testing strategy

> "I follow the test pyramid — many fast unit tests at the base, fewer integration tests in the middle, minimal end-to-end tests at the top. The pyramid exists because slow tests that nobody runs are worse than no tests."

| Level | What It Tests | Speed | Volume |
|-------|--------------|-------|--------|
| Unit tests | Business logic, domain rules, edge cases | < 1s | High |
| Integration tests | DB queries, API contracts, message queue behaviour | 5–30s | Medium |
| Contract tests (Pact) | Consumer-provider API compatibility | ~10s | Key APIs only |
| End-to-end tests | Critical journeys: transfer, login, KHQR payment | Minutes | Minimal |

> "For banking systems specifically, I add **mutation testing**. It verifies that my tests actually catch real bugs — not just achieve line coverage. A test suite that passes when the production code is broken is worse than no tests at all.
>
> The rule I enforce on my team: if a production bug occurs and there was no test covering that scenario, writing that test is part of the fix. It is not optional, and it does not go into a separate ticket."

### Q16 — How do you approach disaster recovery and business continuity?

> "In banking, a two-hour data loss is not a technology problem — it is a legal problem and a reputational one. I design DR before the first line of production code is written, not after the first incident.
>
> Everything starts with two numbers the business must define and own:
> - **RTO (Recovery Time Objective):** how long can the system be down? For payment APIs, typically < 15 minutes.
> - **RPO (Recovery Point Objective):** how much data can we afford to lose? For financial transactions: zero. We cannot lose a single committed transaction.
>
> **RPO = 0** means synchronous replication to a standby database. Every transaction is committed to both primary and replica before the API returns a success response to the client.
>
> **RTO < 15 minutes** means automated failover. If the primary fails, the load balancer redirects to the replica within 60 seconds. The remaining 14 minutes handle application-layer health checks and stakeholder notification.
>
> **The discipline I enforce:** we run DR drills quarterly. A recovery plan that nobody has tested is a wish, not a plan. The drill also reveals whether your RTO is actually achievable — most teams discover it isn't until they test it under pressure."

## 🏦 CMB-Specific Scenarios

These are realistic situations a Director or technical panel may present as case questions.

---

### Scenario A — The Flexcube Timeout

**Prompt:**
> "Our mobile app calls Flexcube for a balance transfer. It takes 8 seconds — sometimes it succeeds, sometimes it times out. Users tap 'Back' and retry. We're seeing duplicate transactions."

**Answer:**

> "This is a classic distributed systems problem and it has a clear fix at two horizons.
>
> **Immediately:** add an idempotency key to every transfer request. The app generates a UUID before sending. If the user retries — or the network retries — the second call hits Redis, finds the key, and returns the original response. Flexcube never sees a duplicate.
>
> **Short-term:** decouple the user experience from the Flexcube call entirely. The app shows 'Transfer submitted — processing' the moment the request is accepted. We process the Flexcube call asynchronously and push the result back via push notification. An 8-second Flexcube response becomes invisible to the user.
>
> **Long-term:** have the conversation with the Flexcube team about why P99 latency is 8 seconds. That's a core banking performance problem that will affect every digital channel — mobile, internet banking, agent banking. It doesn't go away by building around it forever."

---

### Scenario B — KHQR Duplicate Payment

**Prompt:**
> "A customer scanned a QR code and tapped Pay. The network was slow. The app sent the payment twice. The customer was charged twice. The compliance team is flagging it."

**Answer:**

> "**Immediate response:** reverse the duplicate transaction, notify the customer with a clear explanation and timeline for the refund, and file an incident report with compliance. No delay on any of those three.
>
> **Root cause:** the KHQR endpoint lacked idempotency control. Any payment API that can be retried must have deduplication built in.
>
> **Fix:** every KHQR payment request carries a `PaymentRequestID` embedded in the QR code at generation time — unique per payment session. On first receipt, store the ID and outcome in Redis. On any retry within 24 hours, return the original response without re-processing. Flexcube sees the payment exactly once.
>
> **Regression test:** the fix is not complete without a test that reproduces the exact scenario — app sends the same QR payment payload twice, assert exactly one transaction appears in the ledger. That test runs in CI on every deploy going forward."

---

### Scenario C — Salary Day Outage

**Prompt:**
> "It's the 25th. Every corporate client is disbursing payroll through our bulk payment API. The API is returning 503s. 10,000 employees are not getting paid."

**Answer:**

> "**Step 1 — Communicate first.** Notify the executive team, compliance, and corporate banking relationship managers before touching a single log file: 'We have a service disruption affecting bulk payroll processing. We are actively investigating. Next update in 15 minutes.' Silence is not acceptable when employees aren't getting paid.
>
> **Step 2 — Triage fast.** Check the bulk payment API error logs and infrastructure metrics. Is it connection pool exhaustion? Queue saturation? Memory pressure from large payroll files being processed concurrently? The error message tells you.
>
> **Step 3 — Mitigate, not fix.** If it's resource exhaustion, apply temporary rate limiting on bulk submissions, queue the backlog, and let it drain at a sustainable rate. If it's a code bug, roll back to the last known-good version.
>
> **Step 4 — Communicate again, with a commitment.** 'Service restored. Queued payroll will complete processing by 14:00. We will send confirmation to each corporate client.' A specific time, not 'as soon as possible'.
>
> **Post-mortem:** the 25th is not a surprise. Salary disbursement day is predictable every month. We should run a load test at 3x normal volume the week before the 25th, every month. This outage should never happen a second time — and if it does, it means the post-mortem was cosmetic."

---

### Scenario D — NBC Audit

**Prompt:**
> "The National Bank of Cambodia is auditing our digital banking operations next month. They want transaction logs for all KHQR payments in the past six months — including failed and reversed transactions."

**Answer:**

> "If the system is built correctly, this request takes hours to fulfill — not weeks.
>
> I maintain an immutable, append-only audit log in a separate data store from the operational database. Every KHQR payment event — submitted, processing, completed, failed, reversed, corrected — writes a record with: `PaymentID`, timestamp, sender, receiver, amount, currency, status, failure reason, and the user or system that triggered each state change.
>
> For the NBC request, I generate the six-month extract in the format they specify — structured CSV or fixed-width, whichever they require — and sign it with a checksum so they can verify the data hasn't been modified after export.
>
> The deeper point I'd raise in this conversation: if a team cannot respond to a regulator's data request within 48 hours, they don't have a reporting problem — they have a compliance architecture problem. Audit capability should be a design requirement, not something we retrofit after the regulator asks. I build systems that can answer that question before it's asked."
